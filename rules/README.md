# rules

Shared governance artifacts consumed by more than one subsystem. Rule and
metric YAML live here, not inside `dataset/` or `mcp_server/`, because both
subsystems consume the same file:

- The dataset pipeline's Phase 8 codegen (see `docs/implementation_guide.md`)
  turns each YAML into a dbt macro or model at build time.
- The MCP Server's `BusinessRuleResolver` reads the same YAML at runtime.

That shared contract is why a rule change is a single commit that updates
both sides atomically.

## Subdirectories

- `business_rules/` — one YAML per rule; shape mirrors the Business Rules
  sheet of the spec workbook. Canonical reference copies live under
  `spec/generated/business_rules/` and are what the consistency check
  validates against. Hand-authored copies in this directory are the
  pipeline's runtime input once Phase 8 lands.
- `metrics/` — one YAML per KPI; shape mirrors the KPIs sheet. Each file
  carries the canonical formula that Stage 9 renders into a dbt model.

## Shape (for reference until Phase 8 lands)

Business-rule YAML, as exported to `spec/generated/business_rules/`:

```yaml
rule_code: RULE_ED_EXCL_TRIAGE
data_product: clinical_core
name: Exclude triage-only ED encounters
applies_to:
  - clinical_core.encounter
sql_fragment: |
  encounter_type = 'emergency'
  AND (disposition <> 'left_without_being_seen'
       AND TIMESTAMP_DIFF(discharge_ts, admit_ts, MINUTE) >= 15)
```

Metric YAML, as exported to `spec/generated/kpis/`:

```yaml
metric_code: METRIC_LOS_AVG
canonical_formula: |
  AVG(DATE_DIFF(discharge_date, admit_date, DAY))
rules_applied:
  - RULE_LOS_WHOLE_DAYS
tables:
  - encounter
valid_dimensions:
  - facility
  - month
```

Do not edit `spec/generated/` by hand; it is regenerated by
`make spec-export`. Hand-authored copies in `rules/` are the Phase-8 input.
